# ECOCO 社群回覆平台 — 回覆分頁浮動新增鈕 × 故事來源擴充 PRD

**文件版本**：v1.3（現行平台為 v1.1，`prd_reply_helper_inline_edit_compact_list.md` 為 v1.2 待實作項目，本次為另一支功能擴充） **對應 Repo**：`phyto626/ecoco-reply-helper` **狀態**：草案，待確認 **日期**：2026-07-22

## 一、背景與現況問題

本次優化包含兩個獨立但可一起實作的功能，皆源自客服人員日常操作痛點：

| 現況 | 問題 |
| :---- | :---- |
| 「回覆」瀏覽頁沒有任何新增入口，新增回覆範本必須雙擊側邊欄進入「管理後台」分頁，另外找分類、點「新增回覆」 | 客服人員在瀏覽/搜尋回覆時，若發現某個常見問題還沒有範本，得中斷情境切換到後台分頁才能補上，使用頻率高、步驟卻最繁瑣 |
| 社群「故事」收集功能的「來源平台」下拉選單固定為 `IG / FB / Google 地圖 / 其他`，選「其他」時純粹存文字「其他」 | 實際上用戶心得來源不只這四種（如 Threads、Dcard、LINE 社群等），選「其他」無法辨識實際是哪個平台，卡片上看不出差異 |

---

## 二、目標與非目標

### 目標

1. 在「回覆」瀏覽頁新增一顆右下角浮動新增鈕，讓客服在編輯模式下可直接新增回覆範本，不必切到管理後台分頁。  
2. 「故事」來源平台新增固定選項 Threads，並讓「其他」選項可填寫自訂來源名稱，取代單純的「其他」二字。  
3. 兩者皆沿用現有的資料流與彈窗元件，不新增第二套儲存邏輯或資料表欄位。

### 非目標（本期不做）

- 不新增「依平台篩選」故事列表的功能。  
- 不處理「故事」分頁的浮動新增鈕（現行故事分頁工具列按鈕維持原樣，只在解鎖管理模式後顯示）。  
- 不導入登入驗證機制（沿用現行「雙擊解鎖」的軟性防呆）。

---

## 三、使用者故事

1. 身為客服人員，我在「回覆」瀏覽頁看某個分類時，發現少一則常見問題的範本，希望能直接在這一頁把它補上，不用先跑去管理後台重找一次分類。  
2. 身為客服人員，我在故事分頁記錄用戶心得時，來源其實是 Threads 或 Dcard，希望能清楚標記出來，而不是每個非 IG/FB/Google 的來源都混在「其他」裡分不出來。  
3. 身為客服主管，我希望新增功能還是有基本的操作門檻，不要讓一般瀏覽情境下人人都能隨意寫入範本資料。

## 四、功能規格

### 4.1 回覆分頁：浮動新增鈕

- **顯示範圍**：僅「回覆」瀏覽分頁（`switchTab('replies')`），不出現在故事、AI、機台狀態等其他分頁。  
- **顯示條件**：與現行「編輯模式」（`editModeUnlocked`，雙擊側邊欄「回覆範本」解鎖）綁定——只有編輯模式開啟時才顯示浮動鈕；一般瀏覽狀態下不顯示，避免誤觸。  
- **位置與樣式**：畫面右下角固定定位（`position: fixed; bottom/right`），圓形圖示按鈕，比照現行 `.story-add-btn` 的視覺語言但改為 FAB（Floating Action Button）樣式，懸浮於內容之上、不隨頁面捲動位移。  
- **點擊行為**：直接呼叫既有 `openAddModal()`，開啟現有「新增回覆」彈窗（`replyModal`：分類下拉／情境標題／回覆內文）。  
  - 分類下拉預設值＝使用者目前瀏覽中的分類（`currentCatId`），可手動切換。  
  - 儲存邏輯完全沿用 `saveReply()`，不另寫新的 API 呼叫。

### 4.2 故事來源平台擴充

- **新增固定選項**：下拉選單 `storyModalPlatform` 新增 `<option value="Threads">Threads</option>`，放在 `Google` 之後、`其他` 之前。  
- **徽章樣式**：新增 `.story-platform-badge.threads` 樣式（黑底白字，比照 Threads 官方識別色），並於 `storyCardHTML()` 的 `platClass` 判斷邏輯中加入 `else if (platLower.includes('threads')) platClass = 'threads';`。  
- **「其他」自訂名稱輸入（自動累積標籤庫）**：  
  - 選單選到「其他」時，動態顯示一個文字輸入框（例如 `#storyModalOtherName`），並搭配原生 `<datalist>` 提供自動建議。  
  - 選其他選項（IG/FB/Google/Threads）時，該輸入框隱藏。  
  - **標籤庫來源**：不另存獨立清單，即時掃描既有 `STORIES` 陣列，取出所有「非 IG/FB/Google/Threads」的 `platform` 值，去重後動態產生 `<datalist>` 的 `<option>` 清單，供輸入框自動建議。  
    - 去重比對時做正規化：`trim()` 去除前後空白＋轉小寫比對，只要跟清單中某筆只差在大小寫或空白就視為同一項，不重複列入建議清單。  
    - 建議清單只影響「跳出來的建議選項」，**不會**回頭修改資料庫裡既有故事的 `platform` 原始文字（既有資料保持使用者當初輸入的原樣，不做批次正規化）。  
  - 輸入框允許自由輸入全新名稱（不受限於 `<datalist>` 現有選項），送出存檔後該名稱自然成為未來的建議選項之一，達成「自動累積」。  
  - **儲存規則**：若選「其他」，`saveStory()` 存檔前先檢查自訂名稱輸入框：  
    - 為空 → 阻擋儲存，`showToast('請輸入來源名稱', true)`，不呼叫 API。  
    - 有值 → 直接把使用者輸入的原始文字（不做正規化改寫）寫入 `platform` 欄位，取代「其他」二字，不新增額外欄位。  
  - 徽章顯示上，自訂來源名稱一律套用現行「其他」的灰色樣式（`platClass = 'other'`，因為 `platLower` 不會命中 ig/fb/google/threads 任一關鍵字，邏輯上自動落入 else 分支，不需要額外判斷）。  
- **編輯既有資料的回填**：`openEditStoryModal()` 需補上判斷——若 `s.platform` 不含 ig/fb/google/threads 任一關鍵字，代表這是自訂名稱，下拉選單需帶回顯示為「其他」、同時把 `s.platform` 的值回填進自訂名稱輸入框（否則編輯舊的自訂來源資料時，選單會顯示不到對應項或直接跳掉自訂文字）。

## 五、互動狀態總覽

| 功能 | 狀態 | 顯示內容 |
| :---- | :---- | :---- |
| 回覆浮動新增鈕 | 編輯模式關閉（預設） | 不顯示 |
| 回覆浮動新增鈕 | 編輯模式開啟 | 右下角顯示 FAB，點擊開啟 `replyModal`（分類預設＝目前瀏覽分類） |
| 故事來源選擇 | 選 IG / FB / Google / Threads | 自訂名稱輸入框隱藏，徽章套用對應識別色 |
| 故事來源選擇 | 選「其他」 | 顯示自訂名稱輸入框（必填）＋自動建議清單（過去用過的自訂名稱，去重／忽略大小寫與空白），未填寫時擋下儲存並提示 |
| 故事來源選擇 | 編輯既有自訂來源故事 | 選單顯示「其他」，自訂名稱輸入框回填原本文字 |

## 六、技術與資料規格

- **不新增資料表欄位**：`STORIES` 資料結構的 `platform` 欄位維持單一字串，自訂來源名稱直接寫入此欄位；`replies.json` 結構完全不變。  
- **回覆浮動鈕**：純前端 UI 改動，複用 `openAddModal()` / `saveReply()`，僅需新增 FAB 的 HTML/CSS 與「編輯模式開啟時切換 `.visible`」的顯示邏輯（比照現行 `storyAddBtn` 的 `classList.toggle`）。  
- **故事平台判斷邏輯**：`storyCardHTML()` 內的 `platClass` 判斷需新增 `threads` 分支；`openAddStoryModal()` / `openEditStoryModal()` 需新增自訂名稱輸入框的顯示/隱藏切換與資料回填。  
- **驗證邏輯**：`saveStory()` 存檔前新增一段檢查——當 `storyModalPlatform.value === '其他'` 且自訂名稱輸入框為空字串時，中止儲存並提示。

## 七、驗收標準（Acceptance Criteria）

1. 編輯模式關閉時，「回覆」分頁任何畫面都看不到浮動新增鈕。  
2. 雙擊解鎖編輯模式後，右下角立即顯示浮動新增鈕，切換分類、捲動頁面時按鈕位置不隨內容位移。  
3. 點擊浮動新增鈕開啟「新增回覆」彈窗，分類欄位預設為目前瀏覽中的分類；儲存後 Google Sheets 對應分頁同步更新，管理後台清單也能看到新增項目。  
4. 故事「來源平台」下拉選單可選到 Threads，卡片徽章顯示黑底白字樣式。  
5. 選「其他」會顯示自訂名稱輸入框，並跳出過去用過的自訂來源名稱作為建議（`<datalist>`）；留空儲存會被擋下並顯示提示；輸入清單中已有的名稱（含大小寫/空白差異）不會被視為新項目重複列入建議；填寫後儲存，卡片徽章顯示使用者輸入的原始文字（灰底）。  
6. 編輯一則來源為自訂名稱的既有故事時，彈窗正確顯示「其他」並回填原本的自訂名稱文字，不會顯示空白或錯誤選項。
