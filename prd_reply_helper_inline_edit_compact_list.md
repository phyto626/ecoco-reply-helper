## 一、背景與現況問題

目前「回覆範本」瀏覽頁（`replyCardHTML()` 渲染）與「管理後台」（`renderAdminList()` / `replyModal`）是兩套分離的介面：

| 現況 | 問題 |
|---|---|
| 編輯回覆需先雙擊側邊欄解鎖「管理後台」→ 選分類 → 找到項目 → 點鉛筆 → 跳出 `replyModal` 彈窗 | 客服人員在瀏覽/搜尋回覆時，若發現用字要調整，得中斷情境切換三、四步才能改，體感繁瑣 |
| `.reply-card-body` 預設攤開顯示完整換行內文（`white-space: pre-wrap`），每則卡片高度 60–100px | 一頁能看到的則數有限，掃視、比對多則回覆時要一直滾動 |
| 複製按鈕的內容其實是讀 `data-body`，跟畫面顯示文字量無關 | 目前沒有利用這個特性做「預覽收窄、複製不受影響」的設計 |

---

## 二、目標與非目標

### 目標
1. 客服人員可以**直接在回覆範本瀏覽頁**（含分類頁、搜尋結果頁、釘選區）編輯回覆內容，不需切換到管理後台分頁。
2. 回覆列表預設**收合成單行預覽**，點擊展開查看完整內文，大幅減少單頁高度、提升掃視效率。
3. 兩個功能共用同一套資料流（`SHEETS_API` upsert），不新增第二套儲存邏輯。
4. 保留現有「軟性防呆」精神：非授權情境下不應該任何人都能直接改到範本內容。

### 非目標（本期不做）
- 不導入真正的登入驗證機制（Sheets API 權限問題留待後續獨立評估，見「風險」章節）。
- 不改動管理後台的拖曳排序、新增/刪除分類、公告管理等既有功能。
- 不做多人協作即時同步（例如兩人同時編輯同一則的沖突處理）。

## 三、使用者故事

1. 身為客服人員，我在搜尋「滿倉」找到相關回覆時，想直接點開看完整內容並視需要微調用字，不想為了改一個字跑去後台分頁重找一次。
2. 身為客服人員，我在瀏覽某分類底下 10 幾則回覆時，希望一眼掃過標題和關鍵字就能定位到要的那則，而不是每則都攤開一大段文字。
3. 身為客服主管，我希望「編輯權限」還是有一個開關可以收著，不要讓一般瀏覽情境下人人都看得到編輯按鈕，降低誤觸風險。

## 四、功能規格

### 4.1 全域「編輯模式」開關（沿用並取代現行雙擊解鎖）

- 保留現行雙擊觸發手勢（例如雙擊側邊欄 logo 或「回覆範本」文字），但觸發後不再只是**解鎖「管理後台」分頁**，而是切換一個全域旗標 `editModeUnlocked`。
- `editModeUnlocked = true` 時：
  - 所有 `replyCardHTML()` 渲染出的卡片（含分類頁、搜尋結果、釘選區）都會多顯示一顆「編輯」圖示按鈕。
  - 管理後台分頁維持原樣不受影響，仍可正常使用。
- `editModeUnlocked = false`（預設）：卡片維持乾淨，只有「釘選」「複製」兩個按鈕，一般客服日常操作不受任何干擾。
- 狀態建議存在記憶體變數即可（比照現行 `adminUnlocked`），不需要跨頁籤持久化；重新整理頁面即回到鎖定狀態，維持防呆水準與現況一致。

### 4.2 卡片緊湊列表（摺疊 / 展開）

- `replyCardHTML()` 輸出結構調整為兩層：
  - **摺疊列（預設狀態）**：分類標籤 tag + 內文預覽（首行或前 30 字，超出以 `…` 截斷，套用現有 `hl()` 關鍵字高亮）+ 複製按鈕（＋編輯模式下的編輯圖示）+ 展開箭頭圖示。單列高度壓縮至約 40px。
  - **展開區塊（預設 `display:none`）**：完整換行內文，點擊該列標題區域切換顯示/隱藏，箭頭圖示同步旋轉指示狀態。
- 複製按鈕邏輯完全不變：一律讀 `data-body` 完整內容，摺疊狀態下也能直接複製，不強制先展開。
- 釘選按鈕位置隨版面收窄，建議改為 icon-only（`ti-pin`），並與複製、編輯按鈕排在同一列，維持現有 `togglePin()` 邏輯不動。
- 同一頁允許多列同時展開（不做 accordion 互斥），操作邏輯較單純、跟客服「同時比對兩三則」的實際使用情境吻合。

### 4.3 原地編輯（Inline Edit，取代跳轉彈窗）

- 編輯模式下點擊卡片上的編輯圖示：
  1. 若該列尚未展開，自動先展開顯示完整內文。
  2. 展開區塊的內文由純文字顯示切換為 `<textarea>`；同時標題 tag 也切換為文字輸入框（`<input>`），標題與內文皆可編輯，欄位比照 `replyModal` 現有的 `modalTitleInput` / `modalBody`。
  3. 複製 / 展開箭頭 暫時隱藏，改為「儲存」「取消」兩顆按鈕。
- 「儲存」：
  - 呼叫既有 `saveReply()` 內的 upsert 邏輯（建議抽成共用函式 `upsertReply(catId, replyObj)`，管理後台彈窗與原地編輯都呼叫同一支，避免邏輯分裂）。
  - 成功後只更新該張卡片的 DOM 內容與 `CATEGORIES` 資料，不做整頁重新渲染；沿用現有 `showToast()` 顯示「✅ 回覆已更新並儲存」。
  - 失敗則維持編輯狀態、顯示 `showToast('⚠️ 儲存失敗：' + e.message, true)`，不遺失使用者輸入內容。
- 「取消」：還原成展開前的原始文字（先在進入編輯時把原內容存進暫存變數），不呼叫 API。
- 鍵盤支援：`Ctrl+Enter` 觸發儲存、`Esc` 觸發取消，比照客服高頻操作場景。
- 資料對應：卡片需要能穩定找到自己對應的 `catId` + `reply.id`，建議新增輔助函式 `findReplyById(id)` 走訪 `CATEGORIES` 回傳 `{cat, idx, reply}`，取代目前管理後台靠「傳入時的 idx」定位、陣列增刪後容易對不上的寫法。

## 五、互動狀態總覽

| 狀態 | 顯示內容 |
|---|---|
| 預設（editMode 關閉） | 摺疊列：tag + 預覽 + 複製 + 釘選 + 展開箭頭 |
| 展開（唯讀） | 完整內文 + 複製 + 釘選 + 收合箭頭（＋editMode 開啟時多顯示編輯圖示） |
| 編輯中 | 標題輸入框 + 內文 textarea + 儲存 + 取消 |
| 儲存中 | 沿用 `showToast('儲存中…')` |
| 儲存成功 / 失敗 | 沿用現有 toast 樣式與文案 |

## 六、技術與資料規格

- **不新增資料表欄位**，`replies.json` / Google Sheets 結構（`reply_id / title / body`）維持不變。
- **共用函式重構**：
  - `upsertReply(catId, replyObj)`：抽出目前 `saveReply()` 裡 POST `SHEETS_API` 的段落，供管理後台彈窗與原地編輯共用。
  - `findReplyById(id)`：走訪 `CATEGORIES` 回傳定位資訊，取代目前用 `idx` 定位的方式。
- **渲染策略**：原地編輯儲存成功後，只替換該卡片的 DOM 節點內容，不呼叫 `renderCategory()` / `renderSearch()` 整頁重繪，避免捲動位置跳動、閃爍。
- **相容性**：`replyCardHTML()` 在 `renderHome()`、`renderCategory()`、`renderSearch()`、`renderPinnedSection()` 都有呼叫，需確認四處都支援新的摺疊/編輯結構，且各自的 `data-reply-id` 保持一致才能正確定位。

## 七、驗收標準（Acceptance Criteria）

1. `editModeUnlocked = false` 時，任何分類頁 / 搜尋結果頁 / 釘選區的卡片都看不到編輯圖示，複製與釘選功能不受影響。
2. 雙擊觸發後，所有卡片即時顯示編輯圖示，不需重新整理頁面或切換分類。
3. 卡片預設為摺疊狀態，點擊列可展開/收合完整內文；摺疊狀態下複製按鈕可直接複製完整內容。
4. 點擊編輯圖示可直接在卡片內修改標題（tag）與內文，儲存後：
   - 前端畫面即時反映新內容；
   - Google Sheets 對應分頁資料同步更新（可用既有 `SHEETS_API` 驗證）；
   - 管理後台分頁的清單也應顯示更新後的內容（資料來源一致）。
5. 取消編輯不會呼叫任何寫入 API，且畫面還原成編輯前的文字。
6. 儲存失敗時顯示錯誤 toast，且使用者輸入的內容不會被清空或還原。

## 八、風險與待確認事項

| 項目 | 說明 |
|---|---|
| **權限風險（既有問題，非本次新增）** | Google Sheets Apps Script 目前無驗證機制，任何人取得 Web App URL 皆可寫入；本次把編輯入口從「後台分頁」下放到「所有瀏覽頁面」，等於擴大了誤觸/濫用的接觸面。建議：至少維持「雙擊解鎖」的軟性防呆，中長期評估是否需要 GAS 端加入簡單的 token 驗證。 |
| **多列同時編輯** | 若使用者同時點開兩則卡片進入編輯狀態，需確認彼此的儲存/取消互不干擾（各自綁定自己的 `catId + reply.id`，不共用全域暫存變數）。 |
| **手機/小螢幕適用性** | 目前平台以桌機為主要使用情境（客服座位），若未來有行動裝置需求，摺疊列的按鈕排列（複製/釘選/編輯/展開）需另外評估觸控目標大小。 |
| **標題可編輯範圍已確認** | 本次確認標題（`title` tag）與內文皆開放原地編輯，`4.3` 的編輯區塊需同時渲染標題輸入框與內文 textarea。 |

## 十、依 UI/UX Pro Max Skill 檢核強化的設計規格

參考 `ui-ux-pro-max-skill` 資料庫中 Data-Dense Dashboard 樣式規範、Touch/Forms 準則、Icon 規範，對前面的方案做以下具體化與補強：

### 10.1 緊湊列表的具體數值（對應 4.2）

Data-Dense Dashboard 樣式建議的密度數值，直接套用在摺疊列設計上：

| 項目 | 建議值 | 說明 |
|---|---|---|
| 摺疊列高度 | 約 36–40px | 對應資料庫 `--table-row-height: 36px` |
| 列間距 / grid-gap | 8px | 取代目前 `.reply-list { gap: 10px }`，更緊湊但仍可辨識行與行的分界 |
| 卡片內距 | 8–12px（現行為 18px 20px） | 摺疊狀態用小內距，展開為編輯狀態時可局部放大內距讓 textarea 更好操作 |
| 內文字級 | 12–14px（現行 13px 已符合） | 維持現行字級即可，不需再縮小，避免可讀性下降 |

### 10.2 觸控目標與按鈕間距（對應 4.2 / 4.3 的複製・釘選・編輯按鈕）

- 準則要求可點擊區域最小 **44×44px**，相鄰可點擊元素間至少 **8px** 間距。目前 `.copy-btn` / `.pin-btn` 屬於文字＋icon 的窄按鈕，視覺上可以維持小巧，但建議用 `padding` 撐出至少 44px 的實際熱區（而非縮小視覺尺寸），避免未來若在觸控裝置（如平板）上使用時誤觸。
- 摺疊列右側同時放「複製、釘選、編輯、展開箭頭」四個可點擊元素時，彼此間距抓 8px 以上，避免像現行 `.admin-item-actions { gap: 4px }` 這樣過緊。

### 10.3 儲存回饋（對應 4.3 原地編輯）

- 準則要求「Submit Feedback」須有 loading → success/error 的明確回饋，不能點了沒反應。現行 `showToast('儲存中…')` → 成功/失敗 toast 的模式**已符合**此準則，原地編輯延用同一套即可，不需另外設計。

### 10.4 Icon 風格一致性

- 現行專案使用 Google Material Symbols Outline（`material-symbols-outlined`），與準則建議的「統一使用 outline 風格 icon、避免用 emoji 當功能性 icon」一致，新增的編輯／展開箭頭圖示沿用同一套字型即可，不需引入新的 icon 套件。
- 提醒：回覆內文本身包含 emoji（如「💚」「😊」）是**文案內容**、不是 UI 功能圖示，不受此準則限制，維持現狀即可。

### 10.5 新增建議：批次操作（管理後台可選增強，非本期必做）

準則中的「Bulk Actions」提到逐則編輯／刪除太耗時，建議提供多選＋批次動作列。這與本次前台原地編輯的範疇不同，但可以記錄為**管理後台的後續優化方向**：例如未來若要一次刪除或搬移多則過期回覆，可在 `renderAdminList()` 加上勾選框＋頂部批次動作列，避免現行一則一則點刪除按鈕。本期不實作，僅列為待評估項目。
